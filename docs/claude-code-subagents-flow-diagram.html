Core rules

1. Freeze contract before fan-out

For cross-domain work, first create a versioned contract covering routes, DTOs, auth, errors, DB semantics, config, deployment order, and acceptance criteria.

2. Parallel across domains

Data, Backend, Angular, WPF, Mobile, and DevOps lanes may run in parallel after the shared contract is frozen.

3. Sequential inside one domain

For one domain: designer first, then implementer fan-out, then verifier. Do not run the domain verifier before implementation lands.

4. No silent contract changes

If an agent must change anything used by another domain, it must stop with BLOCKED_CONTRACT_CHANGE.

5. Domain signoff is not final signoff

Each verifier checks its domain. The final integration reviewer checks the assembled system and decides commit readiness.

6. Use repair agents for bounded failures

Compile/test/CI failures go to repair or diagnoser agents. Redesign still goes back to contract/domain designers.

Shared policies to implement

Do not create many duplicated model-specific agents. Keep one agent per role/domain and put reusable rules into shared policy files. The Team Lead classifies task risk; the model-routing policy maps that risk to model/effort.
  • model-routing-policy.md — maps task class/risk to model and effort.
  • contract-change-protocol.md — defines when agents must stop and request Contract v2.
  • agent-output-protocol.md — structured statuses and required fields.
  • progress-ledger-protocol.md — compact state ledger for orchestration and compaction.
  • final-integration-gate.md — commit authority and whole-system checks.
  • contract-template.md — shared API/data/UI/DevOps contract format.
  • domain-task-card-template.md — bounded task format for implementers.
  • contract-change-request-template.md — standard output when a shared contract change is needed.
  • verification-report-template.md — standard verifier output and punch-list format.

Contract change protocol

This is the main protection against parallel subagent drift. An implementer may adjust private implementation details, but may not silently change anything another domain depends on.

Local implementation detail changed?
→ Continue and report DONE_WITH_CONCERNS if there is risk.

Shared contract changed?
→ Stop and report BLOCKED_CONTRACT_CHANGE.

Then orchestrator:
1. Marks task blocked in progress ledger.
2. Pauses affected lanes only.
3. Re-runs Team Lead / contract designer / affected domain designers.
4. Produces Contract v2.
5. Broadcasts Contract v2 to affected agents.
6. Agents continue, rework, or discard stale work.
7. Verifiers check against Contract v2 only.

Implementation checklist

Orchestrator

  • Classify domains and risk.
  • Apply model-routing-policy.
  • Freeze contract before fan-out.
  • Maintain ledger.
  • Pause affected lanes on contract change.
  • Run final integration gate.

Designers

  • Accept contract version.
  • Produce bounded task cards.
  • Declare dependencies.
  • Declare forbidden changes.
  • Emit structured status.

Implementers

  • Work on one task only.
  • Write code and tests.
  • Stay inside task card.
  • Never silently change shared contract.
  • Return status + changed files + validation.

Verifiers

  • Run after domain tasks land.
  • Verify current contract version.
  • Return mapped punch-list.
  • Loop until signoff.
  • Never act as final integration gate.

Repair loop

  • Use build/test resolvers for bounded failures.
  • Use diagnosers for unknown root causes.
  • Escalate to contract change if repair changes a shared seam.

Final gate

  • Check all domains together.
  • Run build/test/E2E/migration/deploy checks.
  • Confirm no stale contract versions.
  • Allow commit only after signoff.
